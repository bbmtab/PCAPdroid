# ADBye — AdGuard-Equivalent Content Blocking Roadmap

> **App name / variant:** **ADBye** (AdBye content filter, built on top of PCAPdroid core).
> Repo: `com.emanuelef.remote_capture`, Java, minSdk 23.
> **Brand asset:** Shield vector (gradient pink → blue with checkmark) — see `icon_bluprint.md`.
> Vector source code committed at `app/src/main/res/drawable/ic_adbye_shield.xml`.
>
> **Goal:** Bring layered AdGuard-parity content filtering to PCAPdroid across:
> 1. **Layer 1** (DNS / hosts / in-process blacklist) — works without MITM, ships today.
> 2. **Layer 2** (SNI early-drop) — read domain from TLS ClientHello, drop at VPN TUN.
> 3. **Layer 3** (HTTPS content filtering) — full AdGuard-rule parsing, cosmetic, scriptlet. Requires the external `com.pcapdroid.mitm` addon's `MitmService` to honor a rules-file IPC field; this repo owns the data path, addon repo owns the parser.
>
> Each layer is independently shippable and falls back gracefully if downstream capability is missing.
>
> **Status:** 🟢 IN PROGRESS — Phase 0 + Phase 1 + 1.b landed through `490d9cdd` (gate-armed + parser + HTTP-Host routing + anchor test). The anchor test then FAILED honestly on CI run `29143647827` (drop got 200, gate did not fire) — root cause DETERMINED FROM CODE 2026-07-12 (H2+H4: gate wired post-parse only, not at SYN). **SYN-time consult fix APPLIED 2026-07-12 (`728f5dea`) then REVERTED 2026-07-12 (~`<revert-commit-sha>`)** — adversarial post-CI analysis (including `recompute_conn_block_cb`'s reset mechanism, verified to be a no-op for adblock-sourced blocks because `blacklisted_domain` is sticky under our flag-clear rules) found the fix introduced a latent cardinal constraint-#2 violation risk reachable on the normal operating condition for shared-CDN-IP paths (Cloudflare/Akamai/Google edge: stale LRU `data->info` ≠ real SNI). The connection's SYN drop is **uncorrectable** once set (capture_vpn.c drops synchronously after `zdtun_lookup`; the only `@@`-consult lives in `check_adblock_sni_rules` itself, gated by `!data->to_block`). Card says: "constraint #2 cardinal violation risk on a real device in the field" > "known, named, already-tested HTTP-timing race." **Closed investigation; adblock remains POST-PARSE-ONLY.** The HTTP timing race (gate fires after the GET is forwarded; server's 200 reaches the app) is the new tracked-open architectural limitation, with a code-level rationale embedded at the revert site in `pcapdroid.c` (a short comment explaining WHY adblock is intentionally not consulted at SYN like malware/firewall above it). **Next step HELD per `#8`**: a diagnostic `(ii)` commit (TEMPORARY native `log_d` of `data->info` at SYN + TEMPORARY CI filter widening of `pcapdroid-native:*`) is staged uncommitted in the working tree, to be committed separately once its data→info-at-SYN question is answered; reverted in its own follow-up. See [Phase 1.b Status → "Closed investigation 2026-07-12 — SYN-time adblock consult reverted and adblock remains post-parse-only"](./plan.md) for the full code-trace verdict and reasoning. The 3 `@Ignore`'d tests remain deliberately skipped for the reload-join + CI-emulator-flap reasons separate from this arc.

So in this CI environment the fix is a no-op — the long-standing post-parse race decides non-deterministically. Annoying but not the real problem.
2. **Latent constraint-#2 violation on real devices, reachable through normal shared-CDN-IP operation.** Code-traced three load-bearing facts (this is what made the finding call-able rather than speculative):
   - `check_domain_block_rules` (applied as `728f5dea`) consults the adblock list **against `data->info`** (the LRU hostname); the post-parse consult feeds `(char*)data->ndpi_flow->host_server_name` (the actual TLS SNI / HTTP Host nDPI parsed). `blacklist_match_sni` + `blacklist_match_sni_allowlist` query the `sni` arg per-input. When the LRU hostname ≠ the real SNI (a `tracker.evil.com` cached for a shared CDN IP that this connection uses to host `googlevideo.com`), the at-SYN consult can make a block decision for a host the post-parse consult would have exempted.
   - The SYN drop at `capture_vpn.c` happens synchronously after `zdtun_lookup` synchronously creates the connection and synchronously invokes `handle_new_connection` → `pd_new_connection` → `check_domain_block_rules`, all before `zdtun_forward`. The ClientHello never transits → `host_server_name` never populates. **The drop is final**, not "fixable later."
   - The only `@@`-exemption path lives in `check_adblock_sni_rules` itself, gated by `!data->to_block`. Once `data->to_block=true`, the post-parse `@@`-consult is suppressed at the entry of that same function. There is no code path that consults `host_server_name` against `sni_allowlist` after a SYN-time block.
3. **The "correction" mechanism is a corridor mirror for the adblock case.** `recompute_conn_block_cb` does have an unguarded `data->to_block = (... || ... || blacklisted_domain)` reset, called every ~250ms via `iter_active_connections` from housekeeping. BUT the only `data->blacklisted_domain = false` write outside the malware-path is in the malware whitelist-reload handler (`pcapdroid.c:1152`), which is gated on the malware-whitelist predicates irrelevant to adblock-flagged connections. So for an adblock-flagged connection the recompute preserves `blacklisted_domain=true` (the sticky flag set at SYN's `data->blacklisted_domain = true` in `check_adblock_sni_rules`) and preserves `to_block=true` → **no correction actually happens.** The skeptic who first surfaced this mechanism was right to question my "nothing resets" framing; the right refined verdict is "an unguarded reset exists at `:1108`, but for adblock-sourced blocks it is a no-op because `blacklisted_domain` has no adblock-side clear path."

**Why this is the right call, not a hedge.** Cardinal constraint-#2 violation risk on a real device, in the field, where Play Store / media CDN / Google push could be silently dropped with no error surfaced to the user, for reasons tied to transient CDN-IP sharing that would be maddening to reproduce from a bug report. The HTTP-timing race the fix was meant to solve is a known, named, already-tested gap with a clear boundary (the anchor test documents exactly when the gate does and doesn't fire). Reverting a working fix to a documented limitation is straightforwardly better than shipping a fix whose failure mode is invisible.

Every UI toggle must have an Espresso E2E test that proves the UI click actually alters the underlying network behavior (e.g., Clicking "Ad Blocking OFF" -> Espresso verifies `adblock_rules.txt` is updated -> Network request to `ads.com` succeeds).
5. **JNI & IPC Verification.** For Layer 2 (SNI) and Layer 3 (HTTPS MITM), testing must include actual packet interception. Mocking the PCAPdroid core or Mocking the MitmService IPC is **STRICTLY PROHIBITED** for final validation.
6. **MITM is opt-in.** Master switch `"pref_global_https_filtering"` must default `false`. CA install + `MitmAddon.needsSetup()` gates the L3 menu.
7. **Hot-reload.** Toggling any of the 6 master switches in the Protection tab must re-merge rules and reload the engine without restarting `CaptureService`.
8. **Automated CI/CD Emulator Testing.** Code integration is gated by automated UI/E2E tests running on a CI/CD emulator (e.g., GitHub Actions with UIAutomator/Espresso).
   - The test must install the APK on the emulator, accept the VPN permission dialog, toggle the Protection switches, and perform real (non-mocked) HTTP requests to verify dropping/blocking behavior.
   - This is the final gate: there is no separate physical-device testing step. Code that passes Unit Tests but fails the Emulator E2E pipeline will be rejected.
   - **`@Ignore` discipline.** A test may only be `@Ignore`d when it exercises capability belonging to a phase named in this document. The `@Ignore` message MUST include (a) the named phase (e.g., `Phase 1`) and (b) the path to the plan subsection that owns that capability (e.g., `see plan.md 'Phase 1 — GUI: Protection Tab'`). Generic justifications (`"future work"`, `"TODO"`, `"not implemented"`, etc.) are not acceptable — they become a soft license to defer E2E coverage indefinitely without explicit user sign-off. Every `@Ignore` block must remain greppable and auditable against the current phase plan.
   - **Decision categories for work inside an active phase.** (i) **Proceed + flag for review** — committable without explicit go-ahead: implementing an already-decided deferral; formalizing a phase boundary that the plan already implies; fixing a false-positive test (e.g. test that would pass via DNS failure rather than via filter behavior). (ii) **Explicit go-ahead required** — any change introducing NEW scope not named in `plan.md`; any change visible in a release artifact beyond the local file (CI workflow, merged manifest, Gradle config); any blast-radius change that touches another phase.
   - **CI status reporting rule.** "Green" on the E2E job is defined as: *the in-scope tests for the active phase pass, and every test that is not in-scope is explicitly listed as `@Ignore`d with a named phase reference*. A green run where N tests are skipped is NOT equivalent to a green run where N+M tests are verified — the distinction must be visible in `progress.md` and surfaced in any status narrative, not buried in commit messages.
   - Use `gh run view --job <job-id>` to check workflow progress.
   9. **Capture/telemetry stays off by default; filtering does not depend on it.** VPN interception and nDPI protocol/SNI detection are shared engine layers that both PCAPdroid's original capture feature and ADBye's filtering (Phases 1–4) depend on. This shared layer cannot be disabled without breaking planned filtering phases and is not a target for removal or "optimization."
   - `CaptureSettings.dump_mode` (raw pcap file / HTTP server / UDP / TCP export) MUST remain defaulted to `NONE`. No phase may silently change this default; any change requires updating this constraint explicitly.
   - Per-packet stats accounting and connection-lifecycle tracking stay active — they are load-bearing for Phase 2 (Resource Protection UID/domain bypass) and Phase 3 (SNI matching), not optional "capture bloat" to strip out.
   - **Open decision, not yet made:** what happens to the inherited "Connections" tab UI (PCAPdroid's live traffic list)? Options: (a) keep as an advanced/debug view, (b) hide behind a developer-options flag, (c) remove from user-facing nav entirely. Whichever agent picks this up MUST NOT decide silently — per constraint #8's decision categories, this is a release-visible UI surface change and requires explicit user sign-off, not "proceed + flag for review."
